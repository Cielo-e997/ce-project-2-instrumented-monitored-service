# Demo Script – Instrumented & Monitored Cloud Service

## Slide 1 — Project Overview

This project implements a small cloud service that is fully instrumented for observability.

The goal was not only to deploy an application, but also to make sure that the system is observable and maintainable in a production-like environment.

The application runs on an EC2 instance and exposes simple API endpoints such as /orders and /health.

To make the system observable, I implemented structured logging, custom metrics, monitoring dashboards, and alerting using AWS CloudWatch.

This setup allows engineers to understand system behavior, detect problems quickly, and investigate incidents using real operational data.



## Slide 2 — System Architecture

The system architecture is intentionally simple but follows production best practices.

The application is deployed on an EC2 instance and runs a small Python Flask service.

This service generates logs and metrics that are collected by the CloudWatch agent.

The observability pipeline works like this:

The application produces structured logs → the CloudWatch agent collects those logs → CloudWatch stores and indexes them.

At the same time, custom metrics are published to CloudWatch so we can monitor application behavior and performance.

All of this data is then visualized through CloudWatch dashboards and used to trigger alarms when thresholds are exceeded.

This architecture ensures that the system is both observable and easier to debug.



## Slide 3 — Structured Logging

A key part of observability is structured logging.

Instead of simple text logs, the application generates logs in JSON format.

This allows the logs to contain multiple fields such as timestamps, request paths, response status, latency, and correlation IDs.

Correlation IDs are particularly important because they allow us to trace individual requests across the system.

Each incoming request generates a unique identifier that appears in all related log entries.

This makes it possible to reconstruct the full lifecycle of a request when troubleshooting issues.

Structured logs also make it much easier to search and analyze events using CloudWatch Logs.



## Slide 4 — Custom Metrics

In addition to logs, the application publishes custom metrics.

Metrics provide a high-level view of system behavior and performance over time.

Examples of metrics generated by the application include request counts, error rates, and request latency.

These metrics are sent to CloudWatch where they can be visualized and used for monitoring.

Metrics are essential because they allow engineers to detect anomalies or performance degradation before users are affected.

They also allow us to analyze trends and understand how the system behaves under different workloads.



## Slide 5 — Monitoring Dashboard

To make the system easier to observe, I created a monitoring dashboard in CloudWatch.

The dashboard displays key system metrics and provides a quick overview of the system's health.

It includes metrics that correspond to the Golden Signals used in site reliability engineering.

These signals help engineers quickly identify potential problems.

The dashboard allows us to visualize request activity, performance metrics, and system utilization in one place.

This greatly reduces the time required to diagnose operational issues.



## Slide 6 — Resource Monitoring

In addition to application metrics, I also monitor infrastructure metrics.

CloudWatch automatically provides metrics for EC2 resources such as CPU utilization.

Monitoring infrastructure is important because many application issues originate from resource limitations.

For example, high CPU usage or memory pressure can lead to slow responses or service failures.

By monitoring both application metrics and system resources, we gain a more complete understanding of system behavior.



## Slide 7 — Alerting System

Monitoring alone is not enough if engineers are not notified when problems occur.

For this reason, I configured CloudWatch alarms.

These alarms continuously evaluate system metrics and trigger alerts when thresholds are exceeded.

For example, an alarm can be triggered when CPU utilization becomes too high or when error rates increase.

This allows engineers to react quickly to potential incidents.

Alerts are an essential part of maintaining reliable production systems.



## Slide 8 — Incident Investigation

Observability becomes especially important when diagnosing incidents.

Using the logs and metrics collected in CloudWatch, it is possible to investigate system behavior and identify the root cause of failures.

For example, by correlating request logs with performance metrics, engineers can determine whether an issue is caused by application errors, high load, or infrastructure limitations.

This approach avoids guesswork and allows decisions to be based on real operational data.

The investigation process typically follows the RED method, which focuses on request rate, errors, and request duration.



## Slide 9 — Notification Mechanism

To complete the alerting system, CloudWatch alarms are connected to Amazon SNS.

SNS acts as a notification service that distributes alerts when an alarm is triggered.

When a metric crosses a defined threshold, the alarm activates and sends a notification through the SNS topic.

Subscribers to that topic receive alerts and can start investigating the issue.

This automated notification mechanism reduces the time required to detect incidents and ensures that problems do not go unnoticed.



## Slide 10 — Key Learnings

This project demonstrates how observability improves the reliability and maintainability of cloud systems.

One of the main takeaways is that observability should be built into systems from the beginning rather than added later.

Structured logging enables better debugging and traceability of requests.

Metrics provide visibility into system performance and usage patterns.

Dashboards offer a clear overview of system health.

Alerts allow engineers to respond quickly to incidents.

Together, these components create a monitoring strategy that helps engineers understand system behavior and operate services more confidently in production environments.

## Slide 11 – Challenges

During this project, one of the main challenges I faced was understanding how all the components connect together. Setting up monitoring and alerting was not just about creating resources, but about making sure they actually reflected the behavior of the application.

Another challenge was debugging issues when something did not work as expected. For example, when alarms were not triggering or metrics were not appearing, I had to go step by step to identify what was missing or misconfigured.

It also took some time to get comfortable with navigating CloudWatch and interpreting the data correctly, especially at the beginning.


## Slide 12 – What I Would Improve

If I had more time, I would focus on improving the testing of the system by simulating more realistic scenarios, such as higher traffic or different types of failures.

I would also refine the dashboard to make it more visual and easier to interpret at a glance, adding more meaningful metrics and possibly organizing them differently.

Finally, I would improve the alerting thresholds to make them more precise, so that alerts are triggered only when truly necessary and not too frequently.
