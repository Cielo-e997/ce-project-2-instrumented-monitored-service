# Instrumented & Monitored Cloud Service

## Project Overview

This project implements a small cloud-based service designed with observability in mind.  
Rather than focusing on building a complex application, the main objective was to create a system that can be monitored, analyzed, and troubleshooted effectively once deployed.

The application runs on an Amazon EC2 instance and exposes simple API endpoints such as `/health` and `/orders`.  
What makes this project interesting is the observability stack built around the application. Structured logging, custom metrics, monitoring dashboards, and automated alerts provide visibility into the behavior of the system while it is running.

The goal of this project is to demonstrate how logging, monitoring, and alerting work together to make cloud systems easier to operate and maintain.

---

## Architecture Overview

The system architecture is intentionally simple but reflects common patterns used in cloud environments.

A client sends requests to the API running on an EC2 instance. The application is implemented as a small Flask service that processes requests and generates both logs and metrics.

Application logs are collected and sent to CloudWatch Logs, where they can be searched and analyzed.  
At the same time, custom metrics are published to CloudWatch Metrics so that performance indicators such as request latency and request counts can be monitored.

CloudWatch dashboards visualize these metrics, providing a centralized overview of system health.  
CloudWatch alarms continuously evaluate the metrics and trigger notifications through Amazon SNS when predefined thresholds are exceeded.

This architecture creates a complete observability pipeline that allows the system to be monitored and troubleshooted efficiently.

---

## Key Features

### Structured Logging
The application produces structured JSON logs instead of plain text logs.  
This allows log events to contain consistent fields such as timestamps, log levels, and request information, making them easier to analyze in CloudWatch Logs.

### Custom Metrics
Custom application metrics are published to CloudWatch.  
These metrics help track system behavior over time, including request counts, latency, and potential errors.

### Monitoring Dashboard
A CloudWatch dashboard provides a visual overview of the system's behavior.  
Metrics such as CPU utilization, request latency, and error rates are displayed together to provide a quick understanding of system health.

### Alerting System
CloudWatch alarms monitor important metrics and trigger alerts when thresholds are exceeded.  
This allows the system to detect potential issues automatically rather than relying on manual monitoring.

### Notification Mechanism
Amazon SNS is used to deliver alert notifications generated by CloudWatch alarms.  
When an alarm is triggered, SNS ensures that the alert can be delivered to subscribers so that the issue can be investigated.

## Deployment

The application is deployed on an Amazon EC2 instance and runs as a Flask web service.

Main endpoints:

- `/health` – used to verify that the application is running
- `/orders` – used to simulate order creation and generate application activity

Once deployed, the application produces logs and metrics that are collected by the observability stack.

Together these services provide full visibility into the behavior of the application.

## Key Learnings

One of the most important lessons from this project is the value of observability when operating cloud systems.

Without logs, metrics, and alerts it becomes very difficult to understand what a system is doing once it is deployed.  
By implementing structured logging, custom metrics, dashboards, and automated alerts, the system becomes much easier to monitor and troubleshoot.

Even in a relatively small application, building observability from the beginning provides much better visibility and makes the system more reliable in practice.
