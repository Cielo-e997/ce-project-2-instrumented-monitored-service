# System Architecture

## Overview

The system implemented in this project follows a simple architecture designed to demonstrate observability concepts in a cloud environment.

The application runs as a Flask API deployed on an Amazon EC2 instance. While the application itself is intentionally small, the surrounding infrastructure focuses on providing visibility into the system through logging, metrics, monitoring dashboards, and alerting mechanisms.

The architecture is designed to make the behavior of the system observable and easier to troubleshoot when issues occur.

---

## Main Components

### Client

A client sends HTTP requests to the API endpoints exposed by the application.  
These requests simulate normal application traffic and generate logs and metrics that can be analyzed.

### Application (Flask Service)

The application is implemented as a small Flask web service running on an EC2 instance.

It exposes the following endpoints:

- `/health` – used to verify that the application is running
- `/orders` – simulates order creation and generates application activity

The application produces structured logs and custom metrics during request processing.

### EC2 Instance

The EC2 instance hosts the application and acts as the runtime environment where the service executes.

It also runs the CloudWatch agent responsible for sending logs and metrics to CloudWatch.

### CloudWatch Logs

Application logs are centralized in CloudWatch Logs.

Structured log entries generated by the application are collected and stored in a CloudWatch log group, allowing them to be searched and analyzed.

### CloudWatch Metrics

Custom metrics generated by the application are published to CloudWatch Metrics.

These metrics track system behavior over time and allow performance patterns to be observed.

### CloudWatch Dashboards

A monitoring dashboard visualizes key system metrics.

The dashboard provides a centralized view of important indicators such as request latency, request volume, and CPU utilization.

### CloudWatch Alarms

CloudWatch alarms monitor important metrics continuously.

When a metric exceeds a predefined threshold, the alarm changes state and triggers a notification.

### Amazon SNS

Amazon SNS is used to distribute notifications triggered by CloudWatch alarms.

This ensures that alerts generated by the monitoring system can be delivered automatically.

---

## Observability Pipeline

The observability flow in the system works as follows:

1. A client sends a request to the API.
2. The application processes the request.
3. Structured logs are generated.
4. Logs are sent to CloudWatch Logs.
5. Custom metrics are published to CloudWatch Metrics.
6. Dashboards visualize system behavior.
7. Alarms monitor metrics and detect anomalies.
8. SNS delivers notifications when alarms are triggered.

This pipeline ensures that the system can be monitored and troubleshooted effectively.
